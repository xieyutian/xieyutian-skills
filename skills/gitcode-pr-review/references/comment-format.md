# 行级评论发布格式规范

本文档定义了审查报告中行级评论和整体评论的格式要求，确保 `pr-comment.py` 脚本正确解析并发布到 GitCode PR。

---

## 一、评论发布规则

脚本会自动将审查报告拆分为两类评论：

| 章节 | 发布方式 | 说明 |
|------|----------|------|
| `### 潜在问题` | **行级评论** | 每个问题单独发布到对应代码行 |
| `### 改进建议` + `### 总体评价` | **整体评论** | 合并为一条 PR 评论 |

---

## 二、问题章节位置

潜在问题必须放在 `### ⚠️ 潜在问题` 或 `### 潜在问题` 章节下。

---

## 二、单个问题格式

每个问题必须包含以下元素：

```markdown
#### 序号。问题标题

```语言：文件路径#L 行号
代码片段
```

**风险等级**: 高 | 中 | 低

**问题描述**: 
问题的详细描述...

**建议**: 
改进建议...
```

---

## 三、代码块格式（最关键！）

代码块必须严格遵循以下格式：

````markdown
```cpp:runtime/src/Interpreter/InterpreterSpecific.cpp#L227
    if (tlData == nullptr) {
        LOG(RTLOG_ERROR, "[Interpreter] IsPendingSafePoint called with null thread_local_data_t");
        return false;
    }
```
````

**格式要求**：
- ` ```语言：文件路径#L 行号`
- 语言：使用小写（`cpp`, `c`, `cj`, `python` 等）
- 文件路径：**相对路径**，从仓库根目录开始（如 `runtime/src/Interpreter/InterpreterSpecific.cpp`）
- 行号：支持以下格式
  - 单行号：`#L227`
  - 行号范围：`#L227-230` 或 `#L227-L230`（脚本自动取首行号发布）
- 代码片段：必须是 PR 中实际修改的代码

---

## 四、风险等级标注

必须明确标注风险等级：
```markdown
**风险等级**: 高
```
或
```markdown
**风险等级**: 中
```
或
```markdown
**风险等级**: 低
```

---

## 五、完整示例

````markdown
### ⚠️ 潜在问题

#### 1. 线程安全问题 - 全局状态初始化

```cpp:runtime/src/Interpreter/InterpreterSpecific.cpp#L50
    interpreter_interface_t* operator->()
    {
        if (initialized) {
            return &interpreterInterface;
        }
        LOG(RTLOG_FATAL, "Usage of uninitialized interpreter interface\n");
        return nullptr;
    }
```

**风险等级**: 高

**问题描述**: 
`initialized` 标志不是原子变量，多线程环境下可能存在竞态条件。

**建议**: 
使用 `std::atomic<bool>` 或 `std::once_flag` 确保线程安全的初始化。

---

#### 2. 错误处理不完整 - 空指针检查

```cpp:runtime/src/Interpreter/InterpreterSpecific.cpp#L227
    if (tlData == nullptr) {
        LOG(RTLOG_ERROR, "[Interpreter] IsPendingSafePoint called with null thread_local_data_t");
        return false;
    }
```

**风险等级**: 中

**问题描述**: 
仅记录错误日志后返回 false，可能导致后续逻辑错误。

**建议**: 
```cpp
if (tlData == nullptr) {
    LOG(RTLOG_FATAL, "[Interpreter] IsPendingSafePoint called with null thread_local_data_t");
    abort();
}
```
````

## 六、常见错误

| 错误 | 正确格式 | 说明 |
|------|----------|------|
| ` ```c:src/main.c#L10` | ` ```c:src/main.c#L10` | 文件路径必须是相对路径 |
| ` **风险等级**：高` | ` **风险等级**: 高` | 使用英文冒号 `:` |
| 缺少风险等级 | 必须标注 | 解析器依赖风险等级识别问题块 |
| 代码块无行号 | `#L 行号` | 行级评论需要行号 |
| 行号不在 diff 范围 | 检查实际 diff | GitCode API 会拒绝无效行号 |
| 使用行号范围 `#L10-15` | `#L10` 或 `#L10-15`（均可） | 脚本自动取首行号，范围格式更清晰 |
| 虚构代码片段 | 必须引用实际代码 | ❌ **严禁虚构不存在的代码** |
| "缺少XXX"类问题 | 作为整体评论 | 缺失代码类问题不适合行级评论 |

## 七、核心原则（强制遵守）

> **重要**：以下原则必须严格遵守，违反将导致评论发布失败或误导读者。

### 原则 1：只能引用实际存在的代码

代码块中的代码片段必须是 PR 中**实际存在的代码**，不允许虚构或杜撰。

**错误示例**：
````markdown
```cj:src/test/secret_store_test.cj
// 没有测试 validateKey 功能   ← 这行代码不存在！
```
````

**正确做法**：
- 使用 `Read` 工具或 `git show` 命令**确认代码实际存在**
- 代码片段必须与文件内容**完全一致**（包括缩进、空格）

### 原则 2："缺少XXX"类问题应作为整体评论

以下类型的问题**不适合行级评论**，应放在「改进建议」章节作为整体评论：

| 不适合行级评论的问题类型 | 原因 |
|------------------------|------|
| 缺少测试用例 | 没有代码可引用 |
| 缺少某个功能 | 没有代码可引用 |
| 缺少文档/注释 | 没有代码可引用 |
| 缺少错误处理分支 | 代码不存在 |
| 缺少配置项 | 代码不存在 |

**正确做法**：
- 在「改进建议」章节描述缺失的内容
- 建议添加具体代码，但不要虚构代码块

### 原则 3：禁止虚构代码片段

**严禁**以下行为：
- 杜撰不存在的代码行
- 添加实际文件中没有的注释
- 修改实际代码的内容

**验证方法**：
```bash
# 发布前验证代码实际存在
git -C "$REPO_DIR" show review-<PR编号>:<文件路径> | head -n <行号> | tail -n 1

# 或使用 Read 工具读取文件确认
```

### 原则 4：行号支持范围格式

GitCode API 只支持单个行号，但本技能的脚本**支持行号范围格式**，会自动取首行号发布评论。

**支持格式**：
```
```cj:src/services/secret_store.cj#L34       # 单行号
```cj:src/services/secret_store.cj#L34-35    # 范围格式（自动取首行 L34）
```cj:src/services/secret_store.cj#L34-L35   # 范围格式（自动取首行 L34）
```

脚本解析时显示：`src/services/secret_store.cj#L34 (范围: L34-L35，自动取首行)`

**推荐使用范围**：当问题涉及多行代码时，使用范围格式更清晰地表达问题边界。脚本会自动提取首行号进行发布。

## 八、发布失败处理

如果行级评论发布失败，常见原因：

1. **行号不在 diff 范围内**
   - 使用 `git diff origin/main pr-<PR 号> -- 文件路径` 确认实际修改的行
   - 确保评论的行号在 diff 输出中

2. **文件路径错误**
   - 使用相对路径（从仓库根目录开始）
   - 检查文件名大小写

3. **代码块格式错误**
   - 确保格式为 ` ```语言：文件路径#L 行号`
   - 使用英文冒号 `:`

4. **PR 有多个提交**
   - 行号可能已变化
   - 使用 `git show pr-<PR 号>:文件路径 | head -n 行号 | tail -n 1` 验证

**失败处理**：
- 如果行级评论失败，整体评论仍然会发布
- 手动在 GitCode PR 页面添加行级评论
- 或者者在整体评论中明确标注文件路径和行号

---

## 九、整体评论格式

`### 改进建议` 和 `### 总体评价` 两个章节会被脚本合并为一个整体评论发布。

**格式要求**：
- 两个章节使用标准的三级标题格式
- 内容可以是列表、表格、引用块等任意 Markdown 格式
- 章节之间不需要特殊分隔符

**示例**：
```markdown
### 改进建议

1. 在 README 中说明环境变量配置
2. 启动时检查必要配置

### 总体评价

| 维度 | 评分 | 说明 |
|------|------|------|
| 安全性 | ⭐⭐⭐ | 存在中等风险问题 |

**评审结论**: 建议修复后合并
```

脚本会自动在评论末尾添加分隔线和来源标识。
