# 审查报告格式规范

代码审查完成后，**必须严格按照以下格式输出审查报告**：

## 报告模板

```markdown
## PR #<PR编号> 代码审查报告

### [!] PR 描述分析

**PR 标题**: [从步骤 4.5 获取]
**作者**: [作者名称]
**源分支**: [分支名] -> **目标分支**: [分支名]

**声称的修改点**（从 PR 描述提取）:
1. [修改点1] - [OK] 已实现 / [X] 未实现 / [?] 部分实现
2. [修改点2] - [OK] 已实现 / [X] 未实现 / [?] 部分实现
...

**涉及文件**: X 个文件
- 新增: X 个
- 修改: X 个
- 删除: X 个

---

### [?] 修改全面性评估

| 评估项 | 结论 |
|--------|------|
| 修改完整性 | [OK]/[!]/[X] [说明] |
| 遗漏项 | [如有遗漏，列出] |
| 改进建议 | [如有] |

---

### [!] 变更概述

简要描述本次 PR 的主要变更内容，包括：
- 变更类型（功能开发/Bug修复/重构/依赖升级等）
- 涉及的文件清单
- 核心改动说明

---

### [+] 优点

列出代码变更中值得肯定的方面，例如：
1. 代码规范性改进
2. 性能优化
3. 安全性增强
4. 可读性提升
5. 测试覆盖等

---

### [-] 潜在问题

#### 1. [问题标题]

```[语言]:[文件路径]#L[起始行号]-L[结束行号]
[相关代码片段]
```

**风险等级**：高/中/低
**问题描述**：详细说明问题原因和影响

**建议**：
- 具体的修复或改进建议

#### 2. [问题标题]

（按相同格式列出其他问题）

---

### [*] 改进建议

列出非必须但推荐的改进项：
1. 建议项1
2. 建议项2
3. ...

---

### [!] 总体评价

| 维度 | 评分 |
|------|------|
| 代码质量 | 5/5 |
| 安全性 | 5/5 |
| 修改全面性 | 5/5 |
| 兼容性 | 5/5 |
| 可维护性 | 5/5 |

**总结**：一句话总结审查结论和合并建议。
```

---

## 代码引用格式

代码块必须包含文件路径和行号信息：

```
```语言:文件路径#L起始行号-L结束行号
代码内容
```
```

**示例**：

```
```c:src/socket_buffer.c#L120-L125
if (readSize > rBufSize) {
    return -1;
}
```
```

---

## 报告输出要求

1. **必须包含所有章节**：PR 描述分析、修改全面性评估、变更概述、优点、潜在问题、改进建议、总体评价
2. **问题分级**：每个潜在问题必须标注风险等级（高/中/低）
3. **代码引用格式**：问题的代码块必须包含文件路径和行号，格式为 ```语言:文件路径#L行号```
4. **修改点验证**：必须对每个声称的修改点标记实现状态（[OK]/[X]/[?]）
5. **评分标准**：
   - 5/5：优秀，无明显问题
   - 4/5：良好，有小问题但不影响功能
   - 3/5：一般，存在需要关注的问题
   - 2/5：较差，存在较严重问题
   - 1/5：差，存在阻塞性问题
6. **语言要求**：全部使用中文输出
